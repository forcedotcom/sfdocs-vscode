Author, Validate, Build, and Preview Locally

Once you create your new feature branch, you are all set to start authoring your docs!

Before you begin, familiarize yourself with the repository folder structure.

Start by creating new Markdown files and adding your content. Use Markdown reference to learn more about supported Markdown syntax for your docs.

Working with TOC files

The TOC (Table of Content) files help with structuring your guide and work as a source for the sidebar navigation. They are in simple and easy YAML (Yet-Another-Markup-Language) format.

We recommend that you name the TOC file as _toc.yaml and store it in your guide folder.

Here's an example of a typical TOC file strucutre:

# Sample TOC file format
- title: Overview # title of your guide topic
  link: overview.md # relative link to your markdown file
- title: Setup Your Local Environment
  link: setup-your-local-env.md
- title: Understand your Doc-set Repository Structure
  link: repo-structure.md,
- title: Create and Publish Your Documentation Project
  link: create-publish-from-scratch.md
  topics: # List of child topics
    - title: Step 1 - Clone the Repository & Create a Branch
      link: 1-clone-create-branch.md
    - title: Step 2 - Author, Build, Validate, and Preview
      link: 2-author-build-validate.md
    - title: Step 3 - Commit and Push Your Changes
      link: 3-commit-push-pr.md
- title: Resources
  link: resources.md

The TOC file is structured in the parent-child tree format. Each node of the tree has a title and link. A node can contain multiple children, defined under topics list.

  • title: title of your guide topic
  • link: relative link to your markdown file
  • topics: list of child topics.

Validate Your Docs

There are two ways to validate the Markdown syntax of your docs locally.

  1. Use the markdownlint syntax linter provided within the VSCode toolkit. However, note that we may not support this linter in the future.

  2. Use sfdocs validate command.

    We are still working validations for sfdocs validate command, and it may not match with what you see in markdownlint. We welcome your feedback.

sfdocs validate command also validates broken links. This links include external urls as well as cross-references to other markdown files and images.

caution

Build Your Docs

When you are ready to send your changes to remote, it's always a good idea to build your docs locally and confirm if they are error free. You also must build your docs the first time, to be able to preview them locally.

Use sfdocs build command to build your docs.

Preview Your Docs Locally

You can preview your documentation changes locally, without merging them. This way you can confirm if your changes looks the same when they get merged.

Use sfdocs Preview for viewing your docs on localhost:3000.

Live Preview

You can also preview your docs live as you type. However, changes in TOC or reusable files aren't supported in the live preview mode.

Use sfdocs Preview -live to view the content changes as you type on localhost:3000.
